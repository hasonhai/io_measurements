Methodology¶

Tools

For disk bandwidth test, we use Fio. Fio is short for Flexible IO, a versatile IO workload generator. It was written to benchmark or verify changes to the Linux IO subsystem by Linux developers. Fio is flexible enough to allow detailed workload setups, and it contains the necessary reporting to make sense of the data at completion. Fio is widely used as an industry standard benchmark, stress testing tool, and for IO verification purposes. Fio isn't tied to any OS. It works on any platform from Windows to HP-UX to Android. There are also native IO engines on Linux, Windows, Solaris, etc. This is a key feature to make the measurement repeatable across platforms.

Fio also supports three different types of output formats. The classic output is the default which dumps workload statistics at the end of the workload. There's also support for a CSV format called Terse, though that's slowly diminishing in favor of a JSON-based output format. The latter is far more flexible and has the advantage of being simple to parse for people and computers. However, JSON format only available in later version of FIO. We prefer the terse format since it's available to more versions of FIO. Especially, it is an advantage on some of the hosts where users are not able to upgrade FIO version.

Test Design

During our measurement, we saw that the bandwidth of random read/write and sequential read/write is very different. To keep the same setting for the test and control the time for each test, we have to consider carefully the size of transfered data, number of parallel threads, number of VMs on one host. If the size is too small, the job finish too quickly, we cannot see the a stable bandwidth of sequential read/write. If the data volume is too big, the random read/write test is take too long to finish. The number of parallel threads also affect a lot on the bandwith (as explained in the result assessment section). We run same number of Virtual Machines. We run Fio on the physical host with different number of Fio threads (from 1 to 16 concurrent threads). Then with the same configuration, we run it from inside different flavors of VM. We also make the comparison with the case that each VM has only one thread but we use many concurrent active VMs on the same host to perform I/O operation. We use identical Operating Systems and identical OS' setting across platforms to make sure that the measurement result is reliable.

Our configuration file for Fio has this structure:

; sample setting of one thread
[sequential-read]
rw=read
size=(4gb/number_of_parallel_threads)
directory=/home/cloud-user/fiodata
ioengine=libaio
direct=1
iodepth=16

Where:

  • rw defines the kind of disk access pattern: read (sequential reads), write (sequential writes), randread (random reads), randwrite (random writes), or mixed. We specially focus on these only 2 basic access patterns which was happen mostly in the context Big Data world: Sequential read and sequential write. A typical example is the Hadoop File System (HDFS) which has only sequential read and sequential write.
  • size is the total size of file io for this job. Fio will run until this many bytes of data has been transferred. We choose the total transfer volume to be 4 GBs. This is large enough to avoid many kinds of buffering from the application layet to the OS layer and then to the hardware layer. In facts, some of our initary tests have show that the smaller of size of data volume can cause some very ridiculous results. For example, on some tests we have repoted aggregated throughput of 300MB/s on a single "Seagate Constellation.2 1TB" hard drive while one of professional harddrive benchmarking website claims that the maximum achivable throughput is just about 115 MB/s (http://www.storagereview.com/seagate_constellation2_and_constellation_es2_hard_drive_review). And this is value is more convincing than the 300MB/s one. The data will be divided to threads equally between parallel threads:
    • 1 thread: transfers entire 4 GBs
    • 2 threads: each thread transfer 2 GBs
    • 4 threads: each thread transfer 1 GBs
    • 8 threads: each thread transfer 512 MBs
    • 16 threads: each thread transfer 256 MBs
  • direct I/O set the value of O_DIRECT in the Linux kernel. I/O operations performed against files opened with O_DIRECT bypass the kernel's page cache, writing directly to the storage. Recall that the storage may itself store the data in a write-back cache (in our case, the storage has 64 MBs of write-back cache). There is no strict guarantee that the function will return only after all data has been transferred.
  • iodepth defines the I/O depth of the system. This defines how many io units to keep in flight against the file. The default is 1 for each file defined in the job discription. This value can be overridden with a larger value for higher concurrency. We use I/O depth of 16. In fact, the default I/O depth in some of the Linux distribution can even be set upto 256 or 512.
  • ioengine is libaio which indicates the application to use AIO subsystem. AIO (Asynchronous Linux IO) help to overlaps processing with I/O Operations. Application can submit (batch) IO without waiting for completion. This benefits physical IO in particular when prefetching data. Besides, It allows to accumulate read or write requests so that the IO subsystem can optimize performance by grouping or reordering the IO request in favor of sequential access and large request. It helps to separate calls for submission and completion indication and pipelines operations to improve throughput. It also helps to improved utilization of CPU and devices.

The two top figures represent the average throughput of each thread in the test, while the bottom ones represent the aggregated throughput from all threads/VMs in the same scenario. With the three scenarios, we have very different behavior. Let me remind you about the three scenarios that we have:

  • Bare metal (BM): we have multiple threads concurrently reading data from or writing data to the disk. All the data was read from or written to a directory placed on the same hard drive.
  • Single VM with multiple threads (SVM-MT): in this scenario, we have selected 5 different VM flavors (5 different VM's specs). For each of the flavor, we ran the same test with the involvement of 1 to 16 threads. We plotted the performance of each flavor in a distinct line.
  • Multiple VMs with a single threads (MVM-ST): in this scenario, we have from 1 to 16 VMs, each VM initiates only one process to read/write data.

The same fixed amount of data was used throughout the test. When there is many concurrent threads or concurrent VMs running in one test, we distributed the data equally to all of them.

For the case of SVM-MT We can easily see that performance of threads inside VM is really low when there are more than two processes active at the same time. We use "active" to prefer to the threads or VMs that are reading/writing data at the time. Some may ask, which one is better "fat" VM or "small" VM? The figures said that, in the context of parallel processes competing for throughput on the same hard drive, "fat" or "small" is not a matter. VM with 1 core is naturally the lowest one, but the VM with 16 cores is not the best one as default. In contrast, its performance is even lower than the VM with 2 cores. However, in gereral SVM-MT form a group with similar performance even the VMs have very different virtualised hardward specs. The aggreated throughput for writing is very good and stable. The throughput of threads inside VM with 8 cores is just a little bit lower than the throughput of bare-metal. The case of low throughput of VM with 16 cores maybe cause by the oversubscription of the host. We do not how many VMs from othe tenant of other users is active at the time. Maybe we can have different result for this case if the host is reserved exclusively for the test.

For bare-metal platform, we can easily see that, for the case of data reading, the overhead is increasing when level of concurency is increasing. This lost in performance may result from the interleaving between processes which causes the head of actuator arm to move more often and inscrease seeking time. We expected that we will see the throughput of thread writing data will be a little lower than reading data. But the results turn out completely unexpected. The aggreated throughput of bare-metal reduce gradually when reading data and lower than the writing test in every case of different concurrency levels. In addition, the writting throughput is very stable and does not show the effection of parallelism.

The most unexpected point is on the plot of MVM-ST. We see that its behavior is very strange and provide some possibilities that may happens inside:

1) We used ssh to start the threads inside each VM in parallel. Due to the handshake of ssh-server and ssh-client at the beginning of the test, our processes may not start at the same time, so the aggregated throughput in this scenario is not the aggregated throughput of parallel process from different VMs but rather the aggregated throughput of mixed serial and parallel processes. To fixed, we write a program to listen on port 3000 of all nodes joining the test. On the control machine, we use a client program to send only the "start" command to all nodes through TCP protocol. But the result is not so different with the previous one. That is why we came to another explanation.

2) By observing the bandwidth detail of each node in each test, We saw that there are two groups of nodes that have similar throughput. The picture above represents which group a node is in. We think the creation of these two groups is because of the actual hard drive that the VMs are executing on. We use the same 16 VMs for all the test for MVM-ST scenario. If you look at the picture, the column represent one specific VM and its order in one test. Then we can explain the number for MVM-ST as follows:

  • case 1 VM only: the bandwidth is normal
  • case 2 VMs: because both of them on the same disk, so the bandwidth is just increase a bit thanks to parallel processes.
  • case 4 VMs: VM 1 and 2 on one disk, VM 3 and 4 on another disk, so bandwidth now is double. But because number of VM is even on each disk so we have Jain Fareness Index still high and throughput of 4 of them similar.
  • case 8 VMs: 5 VMs on 1 disk, and 3 VMs on another so the throughput still higher than bare metal but it started to show the affection of competent processes.
  • case 16 VMs: 5 VMs on 1 disk, and 11 VMs on another, throughput now descreases a lot due to high level of concurrency but still similar to bare-metal since the test ran on 2 disks at the same time. So overhead of virtualization is not displayed.

Currently, we have no way to control which disk an instance is executing on. When there are more and more VMs are active, the probability that two or more VMs are executing on different disks is higher. It will turn out that the result for disk stressed test is not correct any more when they are not competeing on the same disk.

3) One other possible hypothesis is that the hypervisor scheduler is not fair (but we prefer more the second hypothesis), some of the VMs may receive more CPU slot and and finish quicker, then other VMs occupy the "have-just-been" empty slot and finish quickly after that. It create a series of wave of parallel processed happen during the test, which makes the aggregated throughput from this scenario uncomparable. If it is the case, we have no way but to modify the hypervisor's setting or changing the its internal execution.

Jain Fairness Index (JFI) has a range from 0 to 1 which indicates the level of fairness between concurrent threads. If the threads receive equal partition of bandwidth, we would achieve index of 1. If only k of n flows receive equal bandwidth (and others get none), index is k/n. In the worst case, JFI has the index of 0. We plot some figures of JFI for the three platforms with confident interval of 95%. The figures show that threads have very equal throughput except the case of Random Read and Random Write. We notice the reader that the figures show value of JFI from 0.85 to 1.05 for a clear view on the JFI of the system. The value for MVM-ST is not good so we ignored in this current plot.

CPU Utilization¶

For CPU utilization, FIO relies on getrusage(). Getrusage() returns resource usage statistics for the calling thread. The resource usages are returned in the structure pointed to by usage, which has the following form:

struct rusage {
    struct timeval ru_utime; /* user CPU time used */
    struct timeval ru_stime; /* system CPU time used */
    ...
    long   ru_minflt;        /* page reclaims (soft page faults) */
    long   ru_majflt;        /* page faults (hard page faults) */
    long   ru_nswap;         /* swaps */
    ...
    long   ru_nvcsw;         /* voluntary context switches */
    long   ru_nivcsw;        /* involuntary context switches */
};

and contains inside the struct:

  • ru_utime: This is the total amount of time spent executing in user mode, expressed in a timeval structure (seconds plus microseconds).
  • ru_stime: This is the total amount of time spent executing in kernel mode, expressed in a timeval structure (seconds plus microseconds).
  • ru_nvcsw: The number of times a context switch resulted due to a process voluntarily giving up the processor before its time slice was completed (usually to await availability of a resource).
  • ru_nivcsw: The number of times a context switch resulted due to a higher priority process becoming runnable or because the current process exceeded its time slice.

FIO reports the CPU utilization for each thread by (user_time/total_run_time) and (system_time/total_run_time), where:

  • user_time is the duration that the thread spending in user space
  • system_time is the duration the t
  • total_run_time is the duration from the time that thread starts until the time that thread finishs.
    runtime = ts->total_run_time;
    if (runtime) {
        double runt = (double) runtime;

        usr_cpu = (double) ts->usr_time * 100 / runt;
        sys_cpu = (double) ts->sys_time * 100 / runt;
    } else {
        usr_cpu = 0;
        sys_cpu = 0;
    }

In this report, we especially forcus on the the serving time (the time that thread actually uses the CPU) and the waiting time (the time that thread has to wait for the other when its slice is up).

The idea to use fixed data size is to see the increasing overhead when parallelism level increases.

There is not much different between the two virtualized scenarios: Single VM with multiple threads and multiple VM with single thread. CPU spending time is a very important metric since it represent how well the process use the allocated CPU slot. For the same amount of work unit, threads inside VM spend double the time that the same ones use inside the bare-metal environment.

In bare-metal environment, the common thought is that sequential write is always slower than sequential read. To illustrate the overhead in CPU spending time and Thread Running Time between the virtualized platforms and bare-metal platform, we plot the normalization of the CPU spending time and Thread Running Time figure using bare-metal data as baseline. The figures show that for the same of transfered data volume, VMs spend double the time using the CPU.
